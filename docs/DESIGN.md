# Plan: Automate Okta Identity Threat Protection (ITP)

## Context

Automate Okta Identity Threat Protection demonstrations for `yourorg.okta.com`. Three goals:
1. **Trigger genuine session hijacking detection** — real cookie replay across regions, not just admin risk flags
2. **Manage entity risk policies as code** — import/apply pattern like existing SOD risk rules
3. **Quick-mode fallback** — admin risk API for fast policy-action testing without infrastructure

ITP is NOT in the Terraform provider, so this follows the **Layer 2 pattern** (Python scripts + GitHub Actions + JSON config), identical to the existing `import_risk_rules.py` / `apply_risk_rules.py` pattern.

**Validated locally:**
- Entity Risk Policy exists: ID `rst1yr8u11qDt6moD1d8`, one "Catch-all Rule" (no action configured)
- User Risk API works: `GET /api/v1/users/{userId}/risk` returns `{"riskLevel":"NONE"}`
- `pyjwt` and `cryptography` already in `requirements.txt`

## Phase 1: Entity Risk Policy Management (import/apply)

Follow the `import_risk_rules.py` / `apply_risk_rules.py` class-based pattern exactly.

**File: `scripts/import_entity_risk_policy.py`**
- Fetches entity risk policy via `GET /api/v1/policies?type=ENTITY_RISK` (Okta standard Policies API)
- Fetches rules via `GET /api/v1/policies/{policyId}/rules`
- Transforms to config format (strips read-only fields, preserves `_metadata`)
- Saves to `environments/{env}/config/entity_risk_policy.json`
- Pattern: `EntityRiskPolicyImporter` class (mirrors `RiskRuleImporter`)
- CLI args: `--org-name`, `--base-url`, `--api-token`, `--output`

**File: `scripts/apply_entity_risk_policy.py`**
- Reads config from `entity_risk_policy.json`
- Compares with current policy/rules in Okta
- Creates/updates/deletes rules (policy itself is immutable — one per org)
- Supports `--dry-run` and `--delete-removed`
- Pattern: `EntityRiskPolicyApplier` class (mirrors `RiskRuleApplier`)
- CLI args: `--org-name`, `--base-url`, `--api-token`, `--config`, `--dry-run`, `--delete-removed`

**Entity Risk Policy Rule schema** (from Okta Policies API):
```json
{
  "name": "High Risk - Force Logout",
  "status": "ACTIVE",
  "conditions": {
    "riskScore": { "level": "HIGH" }
  },
  "actions": {
    "entityRisk": {
      "actions": ["UNIVERSAL_LOGOUT"]
    }
  }
}
```

Available actions: `UNIVERSAL_LOGOUT`, `NONE`
Available risk levels: `HIGH`, `MEDIUM`, `LOW`

**File: `environments/your-environment/config/entity_risk_policy.json`** (generated by import)

**File: `.github/workflows/itp-entity-risk-policy-import.yml`**
- Workflow dispatch with `tenant_environment` input
- Runs `import_entity_risk_policy.py`, commits to branch, creates PR
- Pattern: mirrors `oig-risk-rules-import.yml`

**File: `.github/workflows/itp-entity-risk-policy-apply.yml`**
- Workflow dispatch with `tenant_environment`, `dry_run` inputs
- Runs `apply_entity_risk_policy.py`
- Pattern: mirrors `oig-risk-rules-apply.yml`

## Phase 2: ITP Demo — Two Modes

### Mode 1: Quick Mode (Admin Risk API)

Sets user risk level directly via the admin API. Instant, no infrastructure needed. Useful for testing policy actions (e.g., does UNIVERSAL_LOGOUT fire when risk goes HIGH?). System log shows `"Admin reported user risk"` — not a real detection.

**File: `scripts/trigger_itp_demo.py`** (supports both modes via `--mode quick|real`)

Quick mode flow:
1. Resolve user email to ID via `GET /api/v1/users/{email}`
2. Set risk via `PUT /api/v1/users/{userId}/risk` with `{"riskLevel": "HIGH"}`
3. Optionally monitor system log for policy evaluation events
4. Optionally reset risk to NONE after demo

CLI:
```bash
python3 scripts/trigger_itp_demo.py \
  --mode quick \
  --user user@yourdomain.com \
  --risk-level HIGH \
  --monitor \
  --auto-reset
```

### Mode 2: Real Mode (Two-Region Cookie Replay)

Performs an actual session hijacking simulation — authenticates as a test user in one AWS region, then replays the session cookie from a different region. Okta's ITP ML model detects the IP/device/geo anomaly and fires a genuine `"Session hijacking detected"` event. This is what a real attacker would do.

```

           │                                   │
```

#### Infrastructure Options for the "Attacker" Side

| Option | Pros | Cons |
|--------|------|------|
| **Lambda in eu-west-1** | Ephemeral, cheap ($0.00/demo), no maintenance, different IP every invocation | Needs deployment, 15min timeout |
| **EC2 in eu-west-1** | Persistent, can run Playwright too, reusable | Costs money if left running |
| **Fargate task** | Ephemeral like Lambda, supports longer runs | More setup than Lambda |

**Recommended: Lambda** for the attacker side. It only needs `requests` (replay a cookie via HTTP), runs in <5 seconds, costs nothing, and gets a fresh IP every invocation.

#### Cookie Relay Mechanism

The IDX cookie needs to move from victim to attacker securely and ephemerally:

| Method | How |
|--------|-----|
| **S3 presigned URL** (recommended) | Victim writes cookie to S3 with 60s TTL presigned URL. Attacker reads it. Object auto-expires via lifecycle rule. Both sides only need AWS creds. |
| **SSM Parameter Store** | Victim writes to `/itp-demo/session-cookie` (SecureString). Attacker reads + deletes. |
| **Direct invocation** | Victim calls Lambda with cookie as payload (simplest, no intermediate storage). |

**Recommended: Direct Lambda invocation** — victim EC2 authenticates, captures cookie, then `boto3.invoke(FunctionName='itp-attacker-replay', Payload={'cookie': '...', 'target_url': '...'})`. No intermediate storage, no cleanup, no TTL to manage.

#### Test User Requirements

A dedicated test user for automated demos:
- **Username**: `itp-demo-test@yourdomain.com` (or similar)
- **MFA**: TOTP enrolled, seed stored in SSM Parameter Store (`/your-environment/itp-demo/totp-secret`)
- **Password**: Stored in SSM Parameter Store (`/your-environment/itp-demo/password`)
- **No real access**: User exists solely for ITP demo — no app assignments, no group memberships
- Script generates TOTP codes at runtime using `pyotp` library

#### Real Mode Scripts

**File: `scripts/itp/session_authenticator.py`**
- Runs on the "victim" side (EC2 in us-east-1)
- Uses Playwright to open headless Chromium
- Navigates to `https://yourorg.okta.com`
- Enters username/password (from SSM)
- Handles TOTP MFA challenge (generates code from SSM-stored seed via `pyotp`)
- Extracts IDX cookie from browser context
- Returns cookie value + Okta domain

**File: `scripts/itp/session_replayer.py`**
- Runs on the "attacker" side (Lambda in eu-west-1 or second EC2)
- Receives IDX cookie
- Makes HTTP request to Okta with:
  - The stolen IDX cookie
  - Different `User-Agent` (e.g., Windows Chrome vs macOS Safari)
  - Naturally different IP (eu-west-1 range)
- Returns HTTP response status (confirms cookie was accepted/rejected)

**File: `scripts/trigger_itp_demo.py`** (orchestrator — supports both modes)
- `--mode quick`: Admin risk API (Phase 2, Mode 1)
- `--mode real`: Two-region cookie replay (Phase 2, Mode 2)
  1. Retrieves test user credentials from SSM
  2. Runs `session_authenticator.py` locally or via SSM on victim EC2
  3. Invokes attacker Lambda with the cookie
  4. Monitors system log for genuine session hijacking detection
  5. Optionally resets user risk after demo

CLI:
```bash
# Real mode — full session hijacking simulation
python3 scripts/trigger_itp_demo.py \
  --mode real \
  --user itp-demo-test@yourdomain.com \
  --victim-region us-east-1 \
  --attacker-region eu-west-1 \
  --monitor \
  --auto-reset

# Quick mode — admin risk API
python3 scripts/trigger_itp_demo.py \
  --mode quick \
  --user user@yourdomain.com \
  --risk-level HIGH \
  --monitor
```

#### Lambda Deployment (Attacker Side)

**File: `modules/itp-demo/lambda.tf`** (Terraform module)
- Lambda function in eu-west-1 (or configurable region)
- Python 3.11 runtime
- IAM role with no special permissions (just CloudWatch logs)
- Function code: receives cookie payload, replays against Okta, returns result
- Deployed via the existing `tf-apply.yml` workflow

**File: `modules/itp-demo/lambda/replayer.py`** (Lambda handler)
```python
def handler(event, context):
    cookie = event['cookie']
    okta_domain = event['okta_domain']
    # Replay cookie with attacker User-Agent
    response = requests.get(
        f"https://{okta_domain}/app/UserHome",
        cookies={"idx": cookie},
        headers={"User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) ..."}
    )
    return {"status": response.status_code, "region": context.invoked_function_arn.split(":")[3]}
```

## Phase 3: Event Monitor (shared by both modes)

- Color-coded by severity (risk level, action taken)

| `user.risk.detect` | Risk detected (shows reason: "Session hijacking" vs "Admin reported") |

## API Reference

### Entity Risk Policy
- `GET /api/v1/policies?type=ENTITY_RISK` — list entity risk policies (one per org)
- `GET /api/v1/policies/{policyId}/rules` — list rules for a policy
- `POST /api/v1/policies/{policyId}/rules` — create a rule
- `PUT /api/v1/policies/{policyId}/rules/{ruleId}` — update a rule
- `DELETE /api/v1/policies/{policyId}/rules/{ruleId}` — delete a rule

### User Risk
- `GET /api/v1/users/{userId}/risk` — get current risk level
- `PUT /api/v1/users/{userId}/risk` — set risk level (body: `{"riskLevel": "HIGH"}`)

### System Log (ITP Events)
- `GET /api/v1/logs?filter=eventType eq "user.risk.detect"` — risk detection events
- Key event types: `user.risk.detect`, `policy.entity_risk.evaluate`, `policy.entity_risk.action`, `user.session.end`, `user.authentication.universal_logout`

### Authentication (for Real Mode)
- `POST /api/v1/authn` — primary authentication (username + password)
- `POST /api/v1/authn/factors/{factorId}/verify` — verify TOTP factor
- Session token -> IDX cookie exchange via browser redirect
